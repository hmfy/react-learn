## 项目目标

本项目用于从零学习 React，通过逐步实现一个后台管理系统来掌握核心能力。

主要目标：

- 理解 React 基础（组件、状态、生命周期）
- 学会组件拆分与设计
- 学习如何逐步组织前端项目结构
- 通过实践积累，而不是依赖脚手架或成熟框架

Codex 仅作为辅助工具，而不是主导开发。

---

## 工作方式

在本项目中协助时，请遵循：

- 优先选择最简单可运行的方案
- 保持代码清晰、易读、易理解
- 优先学习价值，而不是“高级架构”
- 小步迭代，不做大规模一次性生成
- 遵循当前项目已有风格

避免过早优化和过度设计。

---

## React 使用原则

- 使用函数组件（Function Component）
- 使用 Hooks，避免 Class 组件
- 每个组件只做一件事（单一职责）
- 当组件变复杂时，拆分组件

默认使用最基础、最直观的 React 写法。

---

## 状态管理

- 优先使用本地状态（useState）
- 必要时再提升状态
- 不主动引入全局状态库（除非明确要求）
- 不为“未来可能用到”提前设计复杂状态

---

## 组件设计

- 尽量可复用，但不要过度抽象
- props 保持清晰、简单
- 避免层层透传 props（props drilling）

优先简单直观的组件结构。

---

## 样式

- 不强制使用任何样式方案
- 遵循当前项目已有方式（CSS / Tailwind 等）
- 样式保持简单、可读
- 避免不必要的样式抽象

---

## 依赖管理

- 不随意引入新库
- 如需引入，需说明理由
- 优先使用 React 原生能力
- 避免引入重量级 UI / 状态库（除非明确要求）

学习阶段：依赖越少越好。

---

## 代码生成规则（重要）

当协助生成代码时：

- 不要一次生成大量代码
- 优先分步骤实现
- 保留已有可运行代码
- 不修改无关文件
- 不写“看起来很完整但不可运行”的代码
- 优先写可运行版本，而不是“架构版本”

---

## 解释策略

必要时简要说明：

- 做了什么修改
- 为什么这么做
- 有哪些替代方案
- 这一步应该学到什么

避免冗长讲解，保持工程化表达。

---

## 学习导向

优先提供：

- 清晰的组件结构
- 明确的 props 使用
- 简单的事件处理
- 易理解的状态更新
- 易理解的副作用处理（useEffect）

如果有多种方案，优先最容易理解的。

---

## 避免事项

除非明确要求，避免：

- 复杂目录结构
- 企业级架构（DDD / Clean Architecture 等）
- 过度抽象
- 过多自定义 hooks
- 滥用 Context
- 过早性能优化
- 复制成熟后台框架设计（如 antd admin）

同时避免：

- 超大组件（难维护）
- UI 与业务逻辑严重耦合
- 重复代码
- 为了省几行代码引入新库

---

## 需求不清晰时

如果需求模糊：

- 优先提供一个最小可运行实现
- 保证易扩展
- 必要时再询问澄清

优先选择“最利于学习”的方案。

---

## 输出优先级

默认输出顺序：

1. 可运行代码
2. 清晰结构
3. 小幅优化
4. 再考虑抽象

不要一开始就给“最复杂/最完整”的方案。

---

## 成功标准

好的代码应满足：

- 新手能看懂
- 结构清晰
- 可以直接运行
- 易于下一步扩展

本项目的标准不是“最专业”，而是：清晰、正确、可学习
